## 1. Strategic items

### 1.1 22× performance gap
**Observation**: 1,000 peptides take ~1 minute on Said's local terminal (TANGO + S4PRED, no UI) but ~22 minutes on the production VPS (Hetzner CX33).

**Hypothesis stack** (ordered by suspected impact):
1. **Uvicorn workers vs CPU contention** — 2 workers × 4 threads on a 4 vCPU box. TANGO is CPU-bound and runs as a subprocess; threads don't help and contend with the event loop.
2. **No batching at the predictor level** — each peptide spawns its own TANGO subprocess. Process-spawn overhead dominates for small peptides.
3. **Vector indexing on the hot path** — ESM-2 embedding writes were moved off the predict path in Wave 2.5 (ISSUE-033) but verify it's still off-path in v0.3.0.
4. **JSON serialization of the response** — per-residue arrays for 1,000 peptides is a lot of JSON. Profile the serializer cost.
5. **Frontend render** — Recharts may be re-rendering on every batch tick, swamping the React main thread.

**Owner**: Said (profile first, then dispatch fix to T2/T3).
**Tooling**: `py-spy record` + Chrome DevTools Performance.

### 1.2 Pre-computed example datasets
**Goal**: zero-wait first impression. "Try example" must render the dashboard in < 2 s.

**Targets**:

**Mechanism**: stub `/api/predict` to recognize `precomputed:<id>` payloads and return the JSON straight from `backend/data/precomputed/*.json`. No TANGO/S4PRED invocation.

**Owner**: T3 (backend route) + T2 (UI "Try example" wiring).

### 1.3 Progressive results loading
**Goal**: for batches > 100 peptides, render rows as they finish instead of waiting for the full job.

**Approach**:
- Server: stream results as Server-Sent Events (SSE) from `/api/predict/stream`. Each event = one peptide normalized payload.
- Client: maintain a `useProgressivePredict()` hook that pushes into datasetStore on every event.
- Backwards-compatible: keep the existing batch `/api/predict` for small jobs.

**Owner**: T3 first (SSE endpoint), T2 second (hook + dashboard wiring).
